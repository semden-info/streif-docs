FIX: Make persistence synchronous with respect to state publication, or add an explicit `flush()`/`awaitIdle()` and call it before any reload/teardown path. The minimal safe fix is to write the append before notifying `listener?.onReveal(...)`.

FILE + locus: `app/src/main/java/no/streif/spike/MainActivity.kt:184-190` (`loadDataThenRun`), `app/src/main/java/no/streif/spike/VisitStore.kt:12-14`  
PROBLEM: Every walk-mode activity creation allocates a new `VisitStore`, and every `VisitStore` creates its own `newSingleThreadExecutor()` that is never shut down. When `TrackingRepository.visitStore` is replaced on the next creation, the old executor thread keeps running until process death. Concrete failure: repeatedly reopening/recreating the activity during testing leaks one extra thread each time, which is exactly the kind of persistent resource leak this prototype is supposed to avoid.  
FIX: Add a `close()` method that shuts down the executor and call it when replacing/discarding a `VisitStore`, or make `VisitStore` process-singleton so only one executor exists.

Overall assessment: the core foreground-service and permission wiring looks mostly defensible for foreground-only location on modern Android, but the state/persistence layer is not safe under real lifecycle churn. The main risk is not cosmetic; activity recreation during an active walk can lose already revealed buildings and break segment continuity, and the async file writer makes that loss observable across restarts.
