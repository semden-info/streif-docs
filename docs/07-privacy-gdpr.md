# Streif — Privacy & GDPR (Приватність і GDPR)

> **Статус:** Чернетка v0.3 — 2026-06-26 (оновлено під on-demand D24; узгоджено з `DECISIONS.md`)
> **Спирається на:** `01`, `02`, `05`, `06`, `09`
> **Призначення:** приватність за дизайном, мінімізація даних, обов'язки за GDPR / personopplysningsloven, DPIA. Норми звірені (червень 2026). Технічна постава → `05`.

> **Наскрізна теза:** **device-local MVP = радикальна мінімізація** — у MVP-0/1 **сирий GPS-трек і стан «розкрито» не покидають пристрій** (немає акаунта, логіну, серверного sync). **Виняток (з 2026-06-26, D24):** для карти йдуть мережеві запити, що розкривають **приблизну** локацію стороннім серверам — тайли базової карти (Kartverket) і **on-demand зони будинків (bbox ~3 км → Cloudflare R2-CDN, фолбек Overpass)**, як у будь-якому онлайн-картографічному застосунку. Це **не** трек і не акаунт, але це задокументований вихідний потік (§7). Складність (згода, профілі, DPIA) приходить лише з соц-шаром `[v2]`.

---

## 1. Принципи приватності

- **Privacy by design & by default** (GDPR Art 25); **мінімізація, обмеження мети й зберігання** (Art 5).
- **Device-local як стратегія** — найкращий захист даних: те, що не зібрано/не передано, не може витекти.
- **Норвезький контекст:** personopplysningsloven = GDPR + нац. доповнення; наглядач — **Datatilsynet**.

## 2. Інвентар даних

| Що | Коли | Де | Примітка |
|---|---|---|---|
| GPS-локація + сирий трек | MVP | **лише локально** | для матчингу; diagnostic-трек ~30 днів локально, тоді видалення (§4) |
| Стан «відвідано» (building_id + час) + статистика | MVP | локально (Room) | результат, не трек |
| **Запит зони будинків (on-demand, D24/D31)** | **MVP** | **→ 3-я сторона** | bbox ~3 км навколо GPS → Cloudflare R2-CDN (дефолт) / Overpass (фолбек); приблизна локація, **не** трек; §7 |
| Запит тайлів базової карти | MVP | → Kartverket | IP + приблизна локація через тайл-координати; §7 |
| Pseudonymous-акаунт (нікнейм; email/OAuth для логіну) | `[v2]` | сервер | реальне ім'я **ніколи** не обов'язкове |
| Агрегований стан для лідербордів; друзі | `[v2]` | сервер | opt-in, взаємна згода |
| Дані активності (Health Connect) | post-MVP | — | особлива категорія (Art 9) |

**У MVP немає акаунта, логіну, серверного sync — стан «розкрито» й сирий трек лишаються на пристрої.** Але карта потребує мережі: тайли бази + on-demand зони будинків розкривають **приблизну** локацію стороннім серверам (§7) — це не персональний профіль, а звичайний онлайн-картографічний потік.

## 3. Правові підстави (Art 6/9)

- **MVP:** обробка локально, під контролем користувача власним інструментом; передачі персональних даних немає. Дозвіл на локацію — поінформована дія користувача.
- **`[v2]`:** соц-функції — **opt-in згода** (Art 6(1)(a)); лідерборд/друзі завжди добровільні, з можливістю відкликати.
- **Health (Art 9, особлива категорія)** — **явна** згода, окремо, при додаванні Health Connect.

## 4. Мінімізація й ретенція

- **Сирий GPS-трек — ніколи на сервер.** Локально: **diagnostic mode** тримає трек + accuracy + matched buildings **~30 днів** (для розбору false positives/negatives на dogfood), потім авто-видалення; ручне видалення будь-коли; у релізі — **налаштування**, за замовчуванням вимкнене (off) або з мінімальною ретенцією. Поза diagnostic — зберігати лише результат «розкрито» + статистику. *(Уточнено після review, `DECISIONS.md` D14.)*
- **Ретенція:** локальні дані живуть, доки користувач хоче; видалення = очистка БД.
- **`[v2]`:** серверні дані — мінімум, визначена ретенція, видалення акаунта стирає все.
- **`[post-MVP]` Агрегована частота візитів** — якщо колись рахуємо community-rarity / «популярні місця» для гейміфікації (`04` §11.5): агрегувати з **мінімальним порогом користувачів на об'єкт**, без можливості де-анонімізації (урок Strava-heatmap 2018 — агрегати розкрили військові бази). Лише з opt-in соц-шаром `[v2]`; узгоджено з device-local-мінімізацією.

## 5. Права суб'єкта (Art 15–22)

- Доступ, виправлення, **видалення**, портативність, заперечення.
- **MVP:** усе локально → повний контроль; «видалити все» очищає локальну БД; експорт (портативність) — опційно.
- **`[v2]`:** видалення акаунта + експорт даних.

## 6. ★ Геолокація (вимога: працювати при згаслому екрані)

- **Підтверджено технічно:** трекінг **триває при згаслому екрані / у кишені / перемиканні застосунків**, доки активний foreground-service (тобто ти стартував прогулянку). Це модель **«while-in-use» + FGS** (`05` §4). Офіційно Android: застосунок «отримує оновлення локації, поки працює foreground service, навіть якщо екран вимкнено».
- **Без `ACCESS_BACKGROUND_LOCATION`.** Межа: ми **не** робимо автоматичного трекінгу з повного фону / без старту користувачем — це «справжній» background (застосунок не відкривали), що потребує цього дозволу + Google Play review + prominent disclosure → **post-MVP, якщо взагалі**.
- **Постійна нотифікація** під час трекінгу — вимога Android FGS і водночас **плюс для прозорості** (користувач завжди бачить, що йде запис).
- **Play-нюанс:** FGS типу `location` ймовірно потребує **декларації Foreground Service** при сабміті (легша за background-location gate, але існує) — підтвердити при публікації.
- **Без real-time location sharing — ніколи** (`01` §8, кейс Strava з військовими базами).
- **Поле (твоя примітка «дивитись під час тестів»):** OEM-killers (Samsung/Xiaomi/Huawei агресивно вбивають FGS — battery optimization), точність/витрата батареї при згаслому екрані, поведінка в Doze. Валідувати на реальних пристроях.

## 7. Треті сторони / потоки даних

- **MVP — тайли базової карти** `topograatone` → HTTP-запити до **Kartverket** (IP + приблизна локація через тайл-координати). Kartverket — держструктура.
- **★ MVP — on-demand зони будинків (D24, нове 2026-06-26):** коли користувач рухається, `AreaLoader` шле запит **з bbox-координатами зони (~3 км навколо GPS)** до зовнішнього сервера будинків:
  - **✅ дефолт (зараз) = Cloudflare R2** (`CdnGeoJsonAreaSource`, `USE_CDN=true`, D31) — контрольована сторона (наш pre-hosted тайл-CDN; OSM+Matrikkelen).
  - **фолбек = Runtime Overpass** (`overpass-api.de` + дзеркала) — стороння OSM-інфра, **не** держструктура; лише dogfood / зони поза покриттям.
  - Передається **приблизна** локація (зона, не сирий трек, не точка); систематично під час прогулянки. Мінімізація: тягнемо грубий тайл і кешуємо (один запит на зону, не на кожен фікс). **Оцінити дата-процесор-умови** для R2/Cloudflare (поточний продакшн-постачальник).
- **`[v2]` / природа:** координати → **api.met.no** (погода), **NVE** (varsom) — передача локації держ-сторонам; врахувати дата-процесор-умови.
- **Принцип:** **жодних рекламних/аналітичних SDK третіх сторін.**

## 8. ★ Datatilsynet / DPIA

- **Правило (порядок перевірки):** (1) якщо обробка є у списку Datatilsynet «завжди-DPIA» → DPIA обов'язковий; (2) інакше — оцінити за **9 критеріями високого ризику EDPB (WP248)**: збіг **≥2 → DPIA**. Для нас: локаційні дані вже дають 1 критерій, тож будь-який другий критерій стає тригером; а систематичний моніторинг публічних зон у великому масштабі робить DPIA безумовним.
- **MVP (device-local): низький ризик** — немає серверної компіляції, немає масштабного моніторингу популяції; це власні дані користувача на власному пристрої. On-demand запит зон (D24) розкриває **приблизну** локацію 3-й стороні, але без зберігання профілю/треку на сервері — не змінює низький профіль ризику (рівень звичайного картографічного застосунку). **DPIA ймовірно не потрібен — але задокументувати оцінку (чому ні), включно з on-demand-потоком і вибором продакшн-постачальника зон.**
- **`[v2]`: повноцінний DPIA перед запуском** — сервер + профілі + лідерборди + потенційно діти (family) → локація + масштаб/діти легко дають ≥2 критеріїв.

## 9. ★ Діти (family-режим, `[v2]`)

- **Вік цифрової згоди в Норвегії — 13** (personopplysningsloven §5; нижній поріг, дозволений GDPR Art 8). Молодше 13 → батьківська/опікунська згода.
- *NB:* пропозиція 2024 підняти поріг до **15** для соцмереж — у публічній консультації; стежити.
- **Family-режим** (реєстрація за дітей) → дитячі дані → батьківський контроль, мінімізація, особлива обережність; ймовірно частина DPIA `[v2]`.

## 10. Безпека даних

- **Локальна БД** — оцінити шифрування at-rest (SQLCipher / Room encryption).
- **Transport** — HTTPS скрізь (api.met.no/Kartverket уже вимагають HTTPS).
- Без зайвих дозволів (принцип найменших привілеїв).

## 11. Прозорість / UX

- **Privacy policy** — проста, bm/nn/en.
- **Контекстне пояснення** «навіщо локація» **перед** системним запитом дозволу.
- **Видима нотифікація** трекінгу (§6).
- Налаштування приватності (що збирається, як видалити).

## 12. Відкриті точки

- **Час DPIA** — перед `[v2]` (оцінку MVP «чому не потрібен» — у цей документ при фіналізації).
- **Чернетка privacy policy** — окремо.
- **Play FGS-декларація** — підтвердити вимогу при сабміті.
- **Шифрування локальної БД** — так/ні (рішення).
- **Вік згоди** — стежити за 13 → 15.
- **Дата-процесор-умови** — Kartverket / MET / **постачальник on-demand зон** (Overpass — лише dogfood; продакшн CC-BY CDN — формалізувати перед релізом, D24/P4).
- **Privacy policy має згадати on-demand-потік** (bbox до сервера будинків) — додати при фіналізації.
