# Streif — Roadmap (Дорожня карта)

> **Статус:** Чернетка v0.2 — оновлено 2026-07-11 (Фаза-0 + Spike-2; враховано D24/D25/D31/D32). ⚠️ **Per-фаза статуси-галочки нижче застарілі** — актуальний стан проєкту → `AGENTS.md` «Поточний статус» + `CHANGELOG.md`; тут — послідовність фаз (план).
> **Спирається на:** `01`–`09`
> **Призначення:** фазований план розробки. **Не календарні дати** (solo-dev ENK, фаза планування), а **послідовність + gate-и**; кожна фаза має вимірний вихід.

---

## 1. Принципи roadmap

- **Fast-first** — найкоротший шлях до тестованого MVP-0; ідеї приходять із реального користування.
- **Найбільший ризик — найраніше** — технічний spike до будь-якого UI.
- **Gate-и між ризикованими переходами** — не йдемо далі, доки попереднє не підтверджено.
- **Вимірний вихід кожної фази** — не «попрацювали», а «отримали X».
- Без жорстких дедлайнів; послідовність важливіша за дати.

## 2. Огляд фаз

| Фаза | Мета | Вихід |
|---|---|---|
| **0 — Gate** | впевнитись у фундаменті | зелене світло на код |
| **1 — Spike** | зняти #1 тех-ризик | рендер + FGS підтверджено |
| **2 — Pipeline** | дані → tiles | on-demand зони (D24; seed `pmtiles` для перф) |
| **3 — Каркас** | карта на пристрої | сіра карта з будинками |
| **4 — Ядро → MVP-0** | петля працює | MVP-0 (dogfood гола механіка) |
| **5 — MVP-1** | приємний продукт | дизайн + гейміфікація |
| **6 — Валідація** | тюнінг/стабільність | стабільний MVP |
| **7 — Реліз** | юр / Play / a11y | готовий до Play |

## 3. ★ Фаза 0 — Gate перед кодом ✅ (пройдено 2026-06-21)

- **Стороннє review повного набору `01–11` + `DECISIONS.md`** (продуктовий зріз; технічний зріз `05–07` — після spike, як домовлено). *(Виконано.)*
- **Адверсивна перевірка оптимальності тех-стеку** — воркфлоу, що кидає виклик кожному ADR з `05` (чи є кращий варіант).
- **Закрити критичні відкриті точки:** ліцензія footprint INSPIRE (з Kartverket), стартові пороги, валідація kommunenummer.
- **Вихід:** рішення «кодимо на цьому стеку» або корективи.

## 4. Фаза 1 — Технічний spike *(історичні маркери; актуальний стан → `AGENTS`/`CHANGELOG`)*

- **Spike-1 v1+v2 — рендер ✅** (D21/D22): MapLibre рендерить кольорові будинки; двошаровий overlay тримає ~60 fps до 4000 visited, async `setGeoJson`.
- **Spike-2 Stage A — трекінг ✅:** реальний FGS `type=location` + Fused + матчинг + збереження + UI + diagnostic mode (D14).
- **Spike-2 Stage B — vehicle-gate ✅** (D5): Activity Recognition + швидкісний гістерезис, польово-валідовано (98% швидких фіксів блоковано).
- **Spike-2 Stage C — battery ✅** (пройдено польовим тестом 2026-07-05, ~6-8%/год екран-off; дозамір на mid-range — опційно post-MVP).
- **Поверх spike (за польовими даними):** D24 on-demand area download (замінив фіксований бандл) ✅; D25 edge-матчинг + closest-approach (замінив центроїд) ✅, польово-верифіковано.
- **Вихід:** #1 ризик (рендер) знято; трекінг-ядро працює. Далі — польовий ретест + Stage C, тоді **незалежне review на межі Spike-2 → MVP-0** (D23). *(`DECISIONS.md` D17/D22/D24/D25.)*

## 5. Фаза 2 — Дані будинків (on-demand, D24)

- **Доставка — on-demand** (не фіксований бандл — замінив стопгап-бандл, `DECISIONS.md` S8): `AreaLoader` тягне зони навколо GPS на льоту + кеш. Тест-джерело = Runtime Overpass; **⚠️ продакшн = pre-hosted CC-BY на CDN (обов'язково перед релізом, D24/P4).** Build-pipeline (Geonorge → Tippecanoe → PMTiles) лишається для seed перф-режимів (`05` §7, `06` §3).
- **Вихід:** будь-яка зона Норвегії довантажується на ходу; продакшн-джерело CC-BY — задача перед релізом.

## 6. Фаза 3 — Android-каркас

- Compose + MapLibre: база `topograatone` + **on-demand будинки** (`AreaLoader`, D24; seed `pmtiles` для перф) + Room-схема. Трекінгу ще немає.
- **Вихід:** застосунок показує сіру карту твоєї місцевості.

## 7. Фаза 4 — Tracking-ядро → MVP-0

- FGS + Fused Location + Activity Recognition (guard авто) + **edge-матчинг + closest-approach** (D25) → Room → зафарбовування overlay + грубий %. *(Ядро трекінгу вже зроблено в Spike-2; на MVP-0 лишається Room-персистенція, eligibility-фільтр D6, onboarding, локалізація, CC-BY-дані.)*
- **UX точності GPS** (D29) — показувати стан надійності GPS, не розкривати/не пропускати тихо. **Захист від ретроактиву** (D30) — апдейти даних не «знімають» уже розкрите (інваріант на Room-персистенції).
- **Вихід:** **MVP-0** — йдеш, будинки фарбуються, прогрес видно; dogfood Денисом (`09` §3).

## 8. Фаза 5 — MVP-1

- **`08`:** фінальні кольори за типами + вторинний сигнал (іконка/патерн toggle); **візуальна мова D28** (контурні іконки колір-за-типом, declutter, collect-unlock без score); **принцип самопояснювальності** (progressive disclosure — будь-яке число/area пояснює себе за ≤2 тапи, `08` §1).
- **`04`:** метрики Coverage/Variety/Discovery, колекції, бейджі, святкування віх, тижневий підсумок. **Coverage-% = per tettsted** (SSB Tettsteder) з видимим кордоном на мапі (**P20** ✅ пайплайн+Android реалізовано 2026-07-11, device-верифікація pending; tettsted-межі + per-tettsted лічильники + boundary-GeoJSON, розширює P18).
- Onboarding ah-ha (+ **deferred auth D26**: пропонувати збереження прогресу після розкриття частини); повна локалізація bm/nn/en.
- **Орієнтація мапи «bearing-up» (P17)** — опційний перемикач, **дефолт north-up**; чиста реалізація через MapLibre `LocationComponent` (напрямлений puck + режими камери, прибирає ручний me-шар/`easeCamera`). Швидкий спайк можливий раніше як експеримент. Відкрите: компасний heading vs GPS-course (`DECISIONS.md` P17).
- **Вихід:** приємний dogfood (`09` §4).

## 9. Фаза 6 — Польова валідація й тюнінг

- Пороги **R/dwell** на місцевості; **OEM-killers** (Samsung/Xiaomi/Huawei); батарея при згаслому екрані; **contrast / colorblind-тести палітри** (`08`); перші **beta-користувачі Sunnmøre**.
- **Прототип «двигуна новизни»** (P10) — **retention-двійка «свіжість» (P19) + «новизна маршруту» (P24)** + **м'яка гейміфікація** (P13): прототипувати й МІРЯТИ retention на beta (мотивує чи тисне?); лишаються proposed до підтвердження даними. Lifetime-«розкрито» не падає (D30).
- **Вихід:** стабільний, відлагоджений MVP.

## 10. Фаза 7 — Підготовка релізу

- **Ліцензія footprint** підтверджена з Kartverket (`06` §11); **Play FGS-декларація** (`07` §6); **privacy policy** (bm/nn/en); **accessibility-аудит** WCAG 2.1 AA; **attribution-екран** (© Kartverket тощо, `06` §9) + **per-image кредит фото POI** (автор + CC-BY-SA, Wikimedia Commons — D34).
- **Вихід:** готовий до Google Play.

## 11. Post-MVP (наброски — без дат, за пріоритетом)

- **v2 соц:** pseudonymous акаунти, локальні лідерборди, друзі — **+ повний DPIA** (`07` §8), Sanctum/backend-sync, FCM.
- **Опційний шар гейміфікації (D33, `04` §11)** — «регулятор глибини» поверх спокійного кору, лише opt-in: **P21** баланс місто↔природа без формули (complementary-колекції + бейдж «Balansert utforsker») · **P22** діорама-2D-first (реальні landmark-типи; справжнє 3D лише після прототипу+retention) · **P23** типи-як-точки-дій · **P25** історичний шар (Riksantikvaren/Nasjonalbiblioteket — біля D27) · **P19+P24** retention-двійка (прототип у Фазі 6). Гарди: rarity ніколи не множник, безпека крос-режимно (`04` §11.3). Герої — відкладено.
- **Дворежимна reveal-система «Місто / Природа» — 2 стовпи, механіка за геометрією (D34).** Кор ambient-reveal + рівноправна «збирай»-механіка (turmål/POI/вершини/хижки з відкритих даних; стежки=Turrutebasen). **Іконка режиму** (тап=ручне перемикання). **Тестуємо обидва режими ASAP** — safety = **release-gate, НЕ test-gate** (тест із проінструктованими знайомими). Черга за складністю: **міські POI** (точка-proximity, без gate) → **вершини/хижки** → **стежки** (лінія-reveal 10%+75%) + **повний safety-GATE** (varsom + yr + disclaimer + SOS; `01` §7). Нова робота = лінія-матчер + рендер ліній/точок + пайплайн на нові типи. Дані-готовність verified 2026-07-12 (Turrutebasen+OSM). Деталі → **D34** / `03` §C-D / `04`.
  - **POI «цікаві місця»** (D27) — read-only з відкритих даних (OSM+Kulturminne+Naturbase+Stadnamn+Wikidata) → особисті локальні нотатки → публічні відгуки (v2); курація гібрид+тірування (declutter). ⚠️ НЕ інжестити куровані бази Stikk UT/MoroTur (sui generis DB-right); turmål-сигнал — через Friluftsrådet-партнерство / відкриті проксі.
  - **Курація-адмінка** — керувати списком POI/turmål (додати/прибрати/позначити, trusted-overlay) → **v2** (Filament-бекенд, D12); для тесту/MVP — файл-список у пайплайні (без бекенду).
- **Авто-маршрут до недослідженого** (P16) — довжина (час/відстань) + кільце/в один бік; обчислювально важко → v2+.
- **Family-режим** (`04` §8, `07` §9); **Strava/Garmin** (Health Connect); **CDN** (bundled→CDN за розміром APK); **розширення регіону** Møre og Romsdal; **повна eligibility / street-corridor** матчинг (точне «виходить на публічну вулицю») — після польового тюнінгу edge-порогів (`DECISIONS.md` D6/D25/P2).

## 12. Наскрізні треки (усі фази)

- **Accessibility** — WCAG 2.1 AA поступово від першого екрана.
- **Приватність** — мінімізація per фаза; DPIA перед v2.
- **Локалізація** — bm/nn/en з MVP-1.
- **Бренд** — назву «Streif» валідувати **до логотипа**: Navnesøk (домен + foretaksnavn + trademark) · тест розуміння на ~10 норвежцях · searchability у сторах (колізія з австрійською трасою Streif) → `DECISIONS.md` P1.

## 12b. UI-міграція на Jetpack Compose + Material 3 (D32)

> Стек залочено (D32). Зараз UI — View-based спайк (програмний FrameLayout). Мета — переписати **лише UI-шар** на Compose + M3, зберігши все нативне ядро. Візуальна мова — D28/`08`.

**НЕ чіпаємо (нижче UI, framework-invariant):** MapLibre MapView (обгортаємо), `TrackingRepository`, `WalkTrackingService`/FGS, `AreaLoader`, Room, Elveg-пайплайн.

**Кроки (послідовно):**
1. **Тема** — `StreifTheme` (Compose M3): кольори-за-типом (D28), типографіка, light/dark; динамічний колір опційно.
2. **Каркас** — `Box`/`Scaffold`: мапа на весь екран + overlay-контроли.
3. **Мапа** — наявний MapView у `AndroidView` + `DisposableEffect` (lifecycle/пам'ять). Перевірений патерн; рендер-ядро (setGeoJson, шари) без змін. НЕ `maplibre-compose` (pre-1.0).
4. **Контроли** — FAB-стек (компас, «до мене»), статус-плашка, filled/extended-FAB «Почати прогулянку» — M3-компоненти (проміжний FAB-фікс 2026-07-05 вже дав вигляд на Views).
5. **Onboarding** — M3 `ModalBottomSheet`/`AlertDialog`.
6. **Стан → ViewModel** — tracking/reveal-стан у `StateFlow`, UI спостерігає (`collectAsStateWithLifecycle`). Це і робить код **CMP-clean** (двері iOS).
7. **Motion** — плавна поява кольору при reveal (M3 motion) — полір, post-паритет.
8. **Прибрати View-код** після паритету.

**⚠️ Window insets (edge-to-edge) — обов'язкова перевірка:** нижні контроли («Почати прогулянку», recenter-FAB) МУСЯТЬ мати `navigationBars`-padding (`Modifier.windowInsetsPadding` / `Scaffold` contentPadding). **Верифікований баг (2026-07-05):** на телефоні з фіксованою always-on 3-кнопковою навігацією (Samsung дружини) кнопка «Почати» частково ховається під навігаційною смужкою (поточний View-UI має жорсткий `bottomMargin=64px`, що не враховує inset). Перевірити на пристрої з always-on nav bar, не лише на Pixel (жести).

**Уточнення (D32):** base M3 `1.4.0` floor + Expressive-компоненти à-la-carte; писати CMP-clean.
**Ризик:** середній, локалізований в UI (ядро не чіпаємо); найтонше — AndroidView-lifecycle мапи (відомий патерн).
**Коли:** ПІСЛЯ польової валідації даних (CC-BY+Elveg) + тюнінгу — щоб не переписувати UI поверх нестабільної механіки.
